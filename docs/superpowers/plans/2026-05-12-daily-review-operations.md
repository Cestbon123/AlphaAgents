# AlphaAgents 每日复盘闭环与实际操作记录计划

日期：2026-05-12

## 背景

当前系统已经具备本地日线选股、候选股持久化、手工持仓维护和持股分析能力，但每日复盘仍主要依赖内存中的候选结果，且缺少“用户实际操作/未操作原因”的结构化记录。这样会导致复盘无法回答“系统给了什么信号、用户做了什么、偏差在哪里、哪些案例值得沉淀”。

## 成功标准

1. 用户可以通过 API 保存某个日期的实际操作记录，服务重启后仍可读取。
2. 每日复盘可以读取最新候选股快照和实际操作记录，即使内存中没有刚跑完的选股结果，也能生成复盘案例。
3. 复盘案例明确展示系统结论、用户动作、偏差、关键原因，并继续生成可沉淀候选。
4. 前端提供一个轻量的实际操作记录入口，并能触发/查看每日复盘结果。
5. 相关后端测试、前端静态检查通过。

## 任务拆分

### 任务 1：实际操作记录模型、存储与 API

- 新增 `OperationRecord` 领域模型。
- 在 SQLite workflow 仓储中新增 `operation_records` 表。
- 新增保存/读取实际操作记录方法，按 `operation_date` 替换当天记录。
- 新增 `/api/v1/review/operations` GET/PUT API。
- 测试覆盖空列表、保存后读取、服务重启后仍可读取。

### 任务 2：每日复盘读取持久化候选和实际操作

- 当内存中没有候选结果时，从最新选股运行快照恢复 `SelectionResult`。
- `ReviewWorkflow` 接受实际操作记录，并将其合并进复盘案例。
- 复盘偏差至少区分：待确认、按系统执行、该买未买、逆系统结论操作。
- 每日复盘继续生成沉淀候选。
- 测试覆盖实际操作记录影响复盘结论。

### 任务 3：前端接入实际操作记录与每日复盘

- 在现有深色工作台中增加实际操作记录区域。
- 候选股列表点击后可以快速带入操作记录。
- 提供保存操作记录按钮。
- 每日复盘运行后展示复盘案例，至少包含股票、系统结论、用户动作、偏差、关键原因。

### 任务 4：验证与文档

- 运行相关 API/工作流测试。
- 运行前端静态测试和 JS 语法检查。
- 更新 `docs/project-status.md`，说明每日复盘闭环的完成范围和仍未完成项。
