# AlphaAgents 沉淀候选确认流程计划

日期：2026-05-12

## 背景

每日复盘已经可以根据候选股快照和实际操作记录生成沉淀候选，但当前沉淀候选仍主要保存在内存中，刷新服务后会丢失，也缺少确认、编辑和废弃入口。MVP 要求“系统生成的沉淀内容必须经用户确认后才可入库”，因此需要补齐人工确认流程。

## 成功标准

1. 每日复盘生成的沉淀候选会写入 SQLite workflow 库。
2. 服务重启后仍可读取沉淀候选。
3. 用户可以修改沉淀候选的标题、内容和状态。
4. 前端沉淀候选区域可以编辑内容、切换状态并保存。
5. 沉淀流程只服务于知识/skill 候选管理，不触发交易执行。

## 任务拆分

### 任务 1：后端持久化与 API

- SQLite workflow 仓储新增 `deposition_candidates` 表。
- 新增保存、读取、更新沉淀候选方法。
- 每日复盘生成候选后同步写入 SQLite。
- Dashboard 优先展示持久化沉淀候选。
- 新增 `GET /api/v1/deposition/candidates`。
- 新增 `PATCH /api/v1/deposition/candidates/{candidate_id}`，支持更新 `title`、`content`、`status`。

### 任务 2：前端确认/编辑入口

- 沉淀候选表格支持编辑内容。
- 每条候选支持状态切换：待确认、已确认、已编辑、已重新生成、已放弃。
- 保存后刷新 Dashboard。

### 任务 3：验证与文档

- 增加后端 API/仓储测试。
- 增加前端静态契约测试。
- 运行完整测试和 JS 语法检查。
- 更新 `docs/project-status.md`。
