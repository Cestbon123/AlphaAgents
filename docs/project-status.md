# AlphaAgents 当前状态

更新时间：2026-05-12

## 当前定位

AlphaAgents 当前仍限定为个人投研、复盘、分析和决策辅助工具，不执行交易、不下单、不接入交易委托能力。当前阶段重点是把本地通达信日线数据、技术指标、K 线观察和规则选股打通，先形成可验证的 MVP 工作台。

## 已完成内容

- FastAPI 后端骨架已建立，包含健康检查、工作流 API、Dashboard API 和本地行情 API。
- 前端已从早期静态页推进到深色工作台形态，包含选股结果、持股分析、沉淀候选、专家配置、运行反馈和 K 线观察区。
- 已接入通达信本地 `.day` 日线文件，支持解析并导入 SQLite。
- 当前验证过的本地数据源路径为 `D:\new_tdx_mock\vipdoc\{sh,sz,bj}\lday`，服务侧通过 WSL 路径读取。
- 2026-05-10 已重新同步本地日线数据，当前 SQLite 行情库最新交易日为 `2026-05-08`。
- 本地行情库已支持行情状态查询、按证券代码读取日线、返回默认技术指标。
- 已从通达信 `T0002/hq_cache/{shs,szs,bjs}.tnf` 解析证券中文名称，并写入本地证券元数据表。
- 后端已实现默认 MACD、KDJ、VOL 指标计算。
- 前端已引入 KLineCharts，并实现离线日 K 线、VOL、MACD、KDJ 副图和“知行趋势线”主图叠加验证。
- KLineCharts 当前以 vendored 静态文件方式离线使用，避免运行时依赖外网 CDN。

## 相比最初 MVP 的主要变化

- 数据源从纯 mock 数据扩展为本地通达信日线数据。
- 选股方向从“示例候选展示”推进为“可由本地日线公式驱动”的规则选股。
- 前端方向从多卡片演示页调整为整体式深色工作台。
- K 线能力从原计划外能力变成当前 MVP 的核心观察组件。
- 自定义公式验证方向明确：优先支持常用通达信公式语义中的 MA、EMA、KDJ、涨跌幅、振幅、板块过滤等规则。

## 当前限制

- 本地日线数据只有 OHLCV，暂未接入完整通达信板块、概念、行业、自定义块数据。
- 当前“剔除创业板、科创板、北证 A 股”先按证券代码前缀和交易所后缀过滤，不等同于完整 `INBLOCK` 数据。
- 全市场选股会先限定常见沪深 A 股代码范围，避免指数、ETF、债券等本地 `.day` 文件进入股票公式筛选。
- 证券名称来自通达信本地缓存文件；若缓存缺失或没有对应代码，会回退显示证券代码。
- 自动选股策略默认剔除名称以 `ST`、`*ST`、`SST`、`S*ST` 开头的标的；按代码单独查看行情或 K 线不受该限制。
- 前端 K 线还处于验证阶段，指标切换、公式管理、指标参数面板尚未完成。
- Dashboard 当前态仍以轻量内存仓库为主；选股运行快照和手动持仓列表已落入 SQLite，持股分析结果、复盘结果等其他业务结果尚未持久化。
- 专家 Skill 仍是规则样例，尚未沉淀成可配置的专家体系。

## 已实现：知行趋势线选股策略

本轮已将“知行趋势线”公式选股落地为独立策略 `ZhixingTrendSelectionStrategy`，工作流会优先使用本地日线库执行策略；当本地库不可用时，开发和测试环境会回退到 mock 示例数据。

公式验证条件：

- `J <= 13`
- `短期趋势线 = EMA(EMA(C,10),10)`
- `知行多空线 = (MA(CLOSE,14)+MA(CLOSE,28)+MA(CLOSE,57)+MA(CLOSE,114))/4`
- `短期趋势线 > 知行多空线`
- `振幅 = (HIGH - LOW) / REF(CLOSE, 1) * 100 <= 4`
- `涨跌幅 = (CLOSE - REF(CLOSE, 1)) / REF(CLOSE, 1) * 100`，范围为 `[-2, 1.8]`
- 剔除创业板、科创板、北证 A 股

该逻辑只生成投研候选，不生成交易指令。后续需要补充完整板块数据时，再把当前前缀过滤替换为真实块数据过滤。

使用当前同步后的本地日线库验证，全市场常见沪深 A 股范围内命中 94 个候选；自动策略默认剔除 ST 标的，单独查看行情或 K 线不受该限制。

## 2026-05-11 新增：可解释选股与运行快照

本轮已把选股结果从“只有候选列表”推进到“可解释、可回看”的形态：

- `SelectionResult` 新增 `strategy_snapshot`，记录策略名称、最新交易日和关键条件明细。
- 知行趋势线策略当前会输出 `kdj_j`、`short_trend_above_long_short`、`amplitude_pct`、`change_pct`、`default_exclusions` 等命中条件。
- 新增 SQLite 选股运行快照仓库，默认路径为 `data/alphaagents-workflows.db`，可通过 `ALPHAAGENTS_WORKFLOW_DB` 覆盖。
- 新增 `GET /api/v1/workflows/selection/runs/latest`，用于读取最近一次选股快照。
- 前端会优先读取最近一次持久化选股结果，并在点击候选股时同步切换 K 线与候选详情。
- 候选详情面板会展示股票、动作、评分、入选原因、命中标签和策略条件明细。

当前仍只记录已入选候选的命中解释；若后续需要排查“为什么某只股票没选上”，需要增加未入选股票的条件诊断接口。

## 2026-05-12 新增：持仓维护与本地行情持股分析

本轮已补齐“手动维护持仓 -> 执行持股分析”的最小闭环：

- 新增 `GET /api/v1/portfolio/positions` 和 `PUT /api/v1/portfolio/positions`，支持本地维护持仓列表。
- 持仓字段包含 `symbol`、`quantity`、`cost_price`、`holding_days`，保存时会整体替换当前持仓列表。
- 持股分析会优先读取手动持仓；如果没有维护持仓，则保留 mock 示例回退。
- 本地日线库可用时，持股分析会用最新收盘价补 `current_price`，用证券元数据补中文名称。
- 前端新增持仓维护面板，可添加、删除、保存持仓；保存后再执行持股分析即可使用最新持仓。

当前仍不维护多账户、多组合，也不接入交易委托。持仓维护只服务于盘后分析和复盘。

## 2026-05-12 新增：实际操作记录与每日复盘闭环

本轮已把“系统信号 -> 用户实际处理 -> 每日复盘 -> 沉淀候选”的最小闭环打通：

- 新增 `OperationRecord` 领域模型，用于记录复盘日期、股票代码、名称、来源、系统结论、用户动作、原因和结果摘要。
- 新增 `GET /api/v1/review/operations` 与 `PUT /api/v1/review/operations`，支持按日期读取和整体替换当天实际操作记录。
- 实际操作记录已落入 SQLite workflow 库；服务重启后仍可读取。
- 每日复盘现在会优先读取最新选股持久化快照，即使内存中没有刚跑完的选股结果，也能基于上次选股结果生成复盘案例。
- 每日复盘会合并实际操作记录，输出系统结论、用户动作、偏差、关键原因，并继续生成沉淀候选。
- 当前偏差判断先覆盖最小可用语义：待用户复盘确认、按系统执行、该买未买、逆系统结论操作。
- 前端新增“复盘记录”面板；点击候选股时会把代码、名称、系统结论和核心原因带入草稿，用户填写实际动作和原因后保存。

边界说明：

- “实际操作记录”只用于投研复盘，不代表交易委托，也不触发任何下单动作。
- 当前实际操作记录、选股快照和复盘案例已经持久化。后续可继续围绕复盘案例补“偏差统计”“错过机会归因”和“周报复盘聚合”。

## 2026-05-12 新增：沉淀候选确认流程

本轮已把“生成沉淀候选 -> 人工编辑/确认 -> 持久化保存”的最小流程补齐：

- 新增 SQLite `deposition_candidates` 存储，保存沉淀候选的 `id`、类型、标题、内容、来源和状态。
- 每日复盘生成的沉淀候选会同步写入 workflow 库，服务重启后仍可读取。
- 新增 `GET /api/v1/deposition/candidates`，用于读取当前沉淀候选池。
- 新增 `PATCH /api/v1/deposition/candidates/{candidate_id}`，支持更新标题、内容和状态。
- Dashboard 会优先展示持久化沉淀候选。
- 前端“沉淀候选”区域支持编辑标题、内容，切换状态并保存。
- 状态沿用 `DepositionStatus`：待确认、已确认、已编辑、已重新生成、已放弃。

边界说明：

- 当前“已确认”仍表示用户确认该沉淀候选可进入后续知识整理队列，尚未生成独立知识库条目或专家 skill 文件。
- “重新生成”状态目前只是人工状态标记，还未接入自动重生成逻辑。

## 2026-05-12 新增：结构化日报

本轮已补齐 MVP 中“每日正式留痕”的最小能力：

- 新增 `DailyReport.report_text`，用于前端展示和复制日报正文。
- 新增 SQLite `daily_reports` 存储，日报按 `report_date` 保存，重复生成会覆盖同一天日报。
- 新增 `POST /api/v1/reports/daily/run`，支持传入 `report_date` 生成指定日期日报。
- 新增 `GET /api/v1/reports/daily/latest`，用于读取最新日报；没有日报时返回 `report: null`。
- 日报会汇总本地行情状态、最新选股快照、持股分析结果、实际操作记录和沉淀候选状态。
- 前端新增“结构化日报”面板，支持选择日期生成日报，并在页面初始化时读取最新日报。

当前日报仍是轻量文本化汇总，适合作为盘后留痕和后续复盘入口；后续可继续扩展为可导出 Markdown/HTML、关联历史 K 线截图、纳入每周复盘统计。

## 2026-05-12 新增：复盘案例持久化

本轮已把每日复盘生成的 `ReviewCase` 从运行时内存推进到可回看的持久化结果：

- 新增 SQLite `review_cases` 存储，按 `review_date` 保存复盘案例。
- 每日复盘运行后会同步保存复盘案例，并在返回结果中包含 `review_date`。
- 新增 `GET /api/v1/review/cases`，支持按 `review_date` 查询。
- 新增 `GET /api/v1/review/cases/latest`，用于读取最新复盘案例。
- Dashboard 新增 `review_cases`，优先返回最新持久化复盘案例。
- 结构化日报的复盘摘要会优先使用持久化复盘案例，展示案例数和偏差分布。
- 前端新增“复盘案例”区域，展示股票、系统结论、实际动作、偏差和关键原因。

当前仍未做跨日期偏差趋势统计；这部分适合放入后续周报复盘和策略诊断模块。

## 2026-05-13 新增：可用初版收口

本轮目标是先形成一版能完整串起核心工作流的可用初版，暂时不把外部高级财务数据、交易数据和自动化下单能力纳入范围。

已完成的收口能力：

- 新增 TdxQuant 元数据导出脚本 `scripts/export-tdxquant-metadata.py`，用于在 Windows 通达信量化环境中导出证券基础信息、市场类别、板块元数据和板块成分关系。
- 新增 TdxQuant 元数据导入脚本 `scripts/import-tdxquant-metadata.py`，将导出的 JSON 写入本地行情 SQLite，形成 `security_profiles`、`sector_metadata`、`sector_members` 等表。
- 知行趋势线选股策略会优先读取真实证券 profile 与板块关系，默认剔除 ST、创业板、科创板、北交所标的；当元数据缺失时仍保留代码前缀兜底过滤。
- 持股分析结果已持久化为最新快照，服务重启后 Dashboard 和结构化日报可以继续读取最近一次持股分析结果。
- 周复盘工作流已从“占位示例”推进为基于持久化复盘案例的聚合统计，输出复盘案例数量、偏差分布、值得沉淀数量和涉及股票数量。
- 前端新增周复盘结果面板，执行“每周复盘”后可以直接看到聚合摘要。
- 新增已确认沉淀条目读取接口 `GET /api/v1/deposition/knowledge-entries`，只返回状态为“已确认”的沉淀候选，为后续专家知识整理和规则沉淀留入口。
- 结构化日报在目标日期没有复盘案例时，会回退使用最新复盘案例快照，避免一键日报因为复盘运行日期与报告日期不同而丢失复盘摘要。

当前可用初版闭环：

1. 同步本地通达信日线数据。
2. 可选同步 TdxQuant 元数据，补齐名称、ST、市值类别/市场类别、板块关系。
3. 执行知行趋势线选股，生成候选、解释、分页列表和 K 线联动。
4. 维护手工持仓，执行持股分析并持久化结果。
5. 记录实际操作，执行每日复盘，生成复盘案例和沉淀候选。
6. 编辑并确认沉淀候选，保留已确认知识入口。
7. 生成结构化日报，执行周复盘聚合。

仍然不包含的边界：

- 不执行交易、不下单、不连接交易委托。
- 不依赖外部实时行情或付费财务数据；当前核心验证仍基于本地通达信日线与 TdxQuant 本地元数据。
- 自定义公式系统还不是完整通达信公式解释器；当前以项目内显式实现的知行趋势线、KDJ、MACD、VOL 和过滤规则为主。
