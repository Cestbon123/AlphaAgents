# AlphaAgents 一期 MVP 需求规格

## 文档状态
- 状态：草案
- 阶段：一期 MVP 需求定义
- 范围：A 股盘后投研主流程
- 说明：本文描述产品需求，不直接约束具体技术实现。

## 产品目标
AlphaAgents 一期要帮助用户围绕既有交易系统，完成从选股、持股分析、复盘到知识沉淀的盘后投研闭环。

系统不执行交易，只提供研究、判断、提醒、复盘和沉淀辅助。

## 核心定位
一期采用流程驱动型工作台。

系统围绕用户的真实日常动作设计：
- 执行选股
- 执行持股分析
- 执行每日复盘
- 执行每周复盘
- 确认知识库或专家 skill 沉淀

专家 skill 是一期核心能力之一。策略负责筛出候选股，专家 skill 负责对候选股和持仓进行二次判断。

## 一期范围
一期只支持 A 股。

一期只做盘后流程，不做盘中实时盯盘。

一期由用户在网页中手动点击执行，不做自动定时任务。

一期需要支持网页仪表盘、结构化日报和可选摘要提醒。

一期需要支持人工确认沉淀。系统可以生成知识库候选或 skill 补充候选，但必须经过用户确认后才能入库。

## 非目标
一期不做自动交易执行。

一期不做盘中实时盯盘。

一期不做自动定时执行。

一期不做策略编写中心。

一期不做回测系统。

一期不做港股、美股、ETF 等多市场支持。

一期不做复杂 skill 版本管理。

一期不做完整知识图谱或复杂知识库治理。

一期不做专家判断量化评分系统。

## 数据来源
行情数据和基本面数据来自券商接口。

候选股由项目内策略模块根据用户既有交易策略自动选出。

持仓由用户手动输入和维护。

实际交易动作由用户在复盘时手动记录或确认。

## 核心页面
### 主工作台
主工作台是用户每日进入系统后的首页。

主工作台需要提供四个独立执行按钮：
- 执行选股
- 执行持股分析
- 执行每日复盘
- 执行每周复盘

主工作台需要展示当日流程运行状态：
- 哪些流程已执行
- 哪些流程未执行
- 最近一次执行是否成功
- 是否存在待确认沉淀项

主工作台需要展示今日摘要：
- 候选股数量
- 买入建议数量
- 待观察数量
- 放弃数量
- 持仓建议数量
- 待确认知识候选数量
- 待确认 skill 补充候选数量

### 选股页面
选股页面用于展示策略选股结果和专家 skill 二次判断。

系统执行选股后，需要展示：
- 候选股列表
- 每只股票命中的选股标准
- 命中理由摘要
- 基本面摘要
- 所属板块热度判断
- 专家 skill 判断理由
- 最终动作标签
- 风险提示

选股阶段的最终动作标签固定为：
- 买入：第二天可根据盘面实际情况择机买入
- 待观察：有潜力，但尚未到买点
- 放弃：从候选中删除

选股阶段不做量化评分。

### 持股分析页面
持股分析页面用于盘后分析当前持仓，并生成第二天操作计划。

用户需要能够手动维护持仓。

系统执行持股分析后，需要展示：
- 当前持仓列表
- 每只持仓的行情摘要
- 每只持仓的基本面摘要
- 专家 skill 判断理由
- 最终动作标签
- 动作理由
- 次日执行提醒
- 风险提示

持股阶段的最终动作标签固定为：
- 继续持有：保持不动
- 放飞：卖出一半或三分之一
- 止损：亏损条件下清仓
- 清仓：通常在盈利后结束本笔交易

### 每日复盘页面
每日复盘页面用于复核当天候选股、实际操作和专家判断质量。

每日复盘需要覆盖：
- 当日候选股中买了哪些
- 当日候选股中没买哪些
- 买入原因
- 未买原因
- 是否存在该买却没买
- 成功案例
- 失败案例
- 错过机会
- 哪些专家判断有效
- 哪些专家判断失效
- 可沉淀内容候选

每日复盘的重点是案例，而不是空泛总结。

### 每周复盘页面
每周复盘页面用于从一周案例中提炼模式。

每周复盘需要覆盖：
- 本周成功案例
- 本周失败案例
- 本周错过机会
- 高频问题模式
- 有效专家判断总结
- 失效专家判断总结
- 建议沉淀项

每周复盘的重点是方法层总结。

### 沉淀确认页面
沉淀确认页面用于管理待确认的知识库候选和 skill 补充候选。

系统生成的沉淀内容不能自动入库。

用户需要能够对每条候选内容执行：
- 确认入库
- 编辑
- 重新生成
- 放弃

一期沉淀能力可以保持轻量，但流程必须完整。

### 专家 Skill 中心
专家 Skill 中心用于让专家能力可见、可追踪、可评估。

一期需要展示：
- 当前可用专家 skills
- 每个 skill 的适用场景
- 每个 skill 最近参与过的判断
- 相关判断后续是否被验证为有效或失效

一期不要求提供复杂 skill 编辑器。

## 核心流程
### 选股流程
用户点击执行选股。

系统拉取当日行情数据和基本面数据。

策略模块根据用户既有交易策略筛出候选股。

专家 skills 对候选股逐只做二次判断。

系统汇总生成选股结果。

选股结果展示在选股页面，并进入结构化日报。

### 持股分析流程
用户维护当前持仓。

用户点击执行持股分析。

系统拉取持仓相关行情数据和基本面数据。

专家 skills 对每只持仓进行盘后判断。

系统输出第二天动作建议。

持股分析结果展示在持股分析页面，并进入结构化日报。

### 每日复盘流程
用户点击执行每日复盘。

系统读取当日候选股、持股分析、实际操作和当日结果。

系统分析买入、未买、该买未买、成功案例和失败案例。

系统评估专家判断有效性。

系统生成知识库候选和 skill 补充候选。

复盘结果展示在每日复盘页面，并进入结构化日报。

### 每周复盘流程
用户点击执行每周复盘。

系统聚合本周每日复盘结果。

系统提炼成功案例、失败案例、错过机会和高频问题模式。

系统总结有效和失效的专家判断。

系统生成更高层级的沉淀候选。

每周复盘结果展示在每周复盘页面。

## 专家 Skill 要求
本文中的专家 skill 指 AlphaAgents 产品内用于投研判断的专家能力，不等同于 Codex 开发环境中的全局 skills。

专家 skill 必须有明确适用场景。

专家 skill 至少需要支持这些场景：
- 选股判断
- 持股判断
- 复盘分析
- 沉淀建议

专家 skill 的输入应尽量使用统一股票上下文。

专家 skill 的输出必须结构化。

专家 skill 输出必须包含理由，不能只给标签。

专家 skill 每次参与判断都需要被记录，便于后续复盘评估。

## 核心数据对象
### 股票上下文
股票上下文是专家 skill 和策略结果汇总的基础输入。

股票上下文应包含：
- 股票代码
- 股票名称
- 所属板块
- 行情摘要
- 基本面摘要
- 板块热度摘要
- 策略命中信息
- 历史画像摘要

### 选股结果
选股结果应包含：
- 股票标识
- 命中的选股标准
- 命中理由摘要
- 专家判断摘要
- 基本面判断摘要
- 板块热度判断摘要
- 最终动作标签
- 核心理由
- 风险提示

### 持股分析结果
持股分析结果应包含：
- 股票标识
- 当前持仓信息摘要
- 当日关键行情判断
- 专家判断摘要
- 最终动作标签
- 动作理由
- 次日执行提醒
- 风险提示

### 每日复盘案例
每日复盘案例应包含：
- 关联股票
- 关联场景
- 当日系统结论
- 用户实际动作
- 结果简述
- 是否存在偏差
- 复盘结论
- 关键原因
- 是否值得沉淀

### 知识库候选
知识库候选应包含：
- 标题
- 结论内容
- 适用场景
- 来源案例
- 来源时间
- 置信说明
- 当前状态

### Skill 补充候选
Skill 补充候选应包含：
- 目标 skill 名称
- 建议补充内容
- 触发场景
- 对应案例来源
- 建议补充原因
- 当前状态

### 运行记录
运行记录应包含：
- 执行类型
- 执行时间
- 输入摘要
- 输出摘要
- 执行状态
- 错误信息
- 关联结果

## 交付形式
### 网页仪表盘
网页仪表盘是第一期主入口，负责承载流程触发、结果查看和待处理事项。

### 结构化日报
结构化日报是每日正式留痕。

日报应包含：
- 当日市场摘要
- 选股结论摘要
- 持股建议摘要
- 当日复盘摘要
- 待确认沉淀项摘要

### 可选摘要提醒
摘要提醒用于提示当日重点结果。

摘要提醒不是完整报告，只提供简短结论和跳转入口。

## 验收标准
一期完成后，用户应该能够手动点击执行选股流程。

系统应该能够基于策略产出 A 股候选股。

专家 skills 应该能够对候选股给出买入、待观察或放弃。

用户应该能够手动维护持仓。

专家 skills 应该能够对持仓给出继续持有、放飞、止损或清仓。

系统应该能够生成每日复盘。

系统应该能够生成每周复盘。

系统应该能够识别成功案例、失败案例和该买未买案例。

系统应该能够生成待确认的知识库候选或 skill 补充候选。

用户应该能够编辑、重新生成、确认或放弃沉淀候选。

核心结果应该能够在网页和结构化日报中查看。

## 后续需求清单
这些需求明确记录，但不进入一期 MVP：
- 策略编写中心
- 回测系统
- 自动定时执行
- 盘中实时盯盘
- 即时交易提醒
- 自动交易执行
- 港股、美股、ETF 等多市场支持
- 复杂 skill 版本管理
- 完整知识库治理体系
- 专家判断量化评分系统

## 待确认问题
- 券商接口的具体类型和可用数据字段需要在架构设计阶段确认。
- 持仓手动输入的字段范围需要在交互设计阶段确认。
- 专家 skill 的存储格式、调用方式和版本策略需要在技术方案阶段确认。
- 结构化日报的具体格式需要在前端和文档设计阶段确认。
