# AlphaAgents Agent 能力基线需求文档

## 1. 背景

AlphaAgents 当前已经具备本地行情数据、a-stock-data 外部数据缓存、策略选股、个股工作台、复盘、研究报告、Agent Chat 和记忆系统等能力。

但从用户视角看，当前最大问题不是缺少单点功能，而是：

- 不清楚 Agent 到底能做什么。
- 不清楚 Agent 回答时使用了哪些数据。
- 不清楚本地数据、a-stock-data、工作流数据、Agent 记忆之间如何串联。
- 不清楚后续如何新增或组织 Agent 的投研能力。

因此，下一阶段的重点不是继续新增孤立功能，而是把 AlphaAgents 作为 Agent 的能力基线完善清楚，让现有能力能够被 Agent 稳定、透明、可解释地编排使用。

## 2. 目标

本阶段目标是建立 AlphaAgents 的 Agent 能力基线。

具体目标：

1. 明确 Agent 的能力边界：它能做什么、不能做什么、哪些操作需要确认。
2. 明确数据使用方式：本地数据、a-stock-data、工作流数据、记忆数据分别在什么场景下使用。
3. 梳理现有功能如何被 Agent 串联：从用户问题到数据查询、工具调用、结果展示、记忆沉淀形成闭环。
4. 定义 AlphaAgents 内部 Skill 的概念和规范，作为后续能力扩展的统一入口。
5. 让用户在 Chat 中能够理解 Agent 的能力、依据和限制，而不是面对一个黑盒聊天框。

## 3. 非目标

本阶段不做以下事情：

- 不新增交易执行、下单、撤单、自动调仓能力。
- 不追求复杂多步自主 Agent 工作流。
- 不引入新的数据供应商。
- 不重构现有数据层、策略层、工作流层。
- 不把所有能力做成新页面。
- 不把 Skill 做成插件市场或复杂编排系统。

本阶段强调“把已有能力组织清楚、用清楚、解释清楚”。

## 4. Agent 定位

AlphaAgents Agent 是个人 A 股投研助手。

它的核心角色不是替用户交易，而是：

- 帮用户查数据。
- 帮用户理解当前状态。
- 帮用户串联历史判断。
- 帮用户发现风险和待观察点。
- 帮用户沉淀复盘和经验。

Agent 输出必须保持以下原则：

- 只做投研、复盘、分析和决策辅助。
- 不输出交易指令或下单建议。
- 涉及不确定性时必须说明数据缺口和假设。
- 涉及写入记录、更新跟踪状态、更新画像时必须先获得用户确认。
- 引用历史判断时必须说明来源、日期或上下文。

## 5. 数据能力基线

### 5.1 数据类型

当前项目内的数据应被分为四类。

| 数据类型 | 来源 | 典型内容 | Agent 中的作用 |
| --- | --- | --- | --- |
| 本地行情数据 | TDX/SQLite | 日线、K 线、成交量、股票名称、板块、市场元数据 | 判断趋势、计算指标、执行选股、展示 K 线 |
| 外部扩展数据 | a-stock-data/cache | 估值、资金流、龙虎榜、公告、新闻、板块补充信息 | 补充基本面、资金面、事件面信息 |
| 工作流数据 | workflow SQLite | 持仓、选股结果、研究报告、日报、复盘案例、操作记录 | 支撑用户日常投研流程和复盘闭环 |
| Agent 记忆 | agent memory tables | 用户画像、历史决策、个股印象、对话历史 | 支撑个性化、历史回顾和上下文延续 |

### 5.2 数据使用原则

Agent 回答时不应只依赖模型常识。凡是问题涉及股票、策略、持仓、复盘或历史判断，应优先使用项目内数据。

基础规则：

1. 问个股时，优先使用本地行情数据和个股工作台数据。
2. 问基本面、资金流、新闻、公告时，优先使用 a-stock-data 缓存或外部数据缓存。
3. 问“我之前为什么这么判断”时，优先使用 Agent 记忆、操作记录和复盘案例。
4. 问“今天/最近有什么关注点”时，优先使用持仓、提醒、最新选股结果、日报和复盘记录。
5. 数据缺失时，Agent 必须明确说明缺失项，而不是用模糊语言补全。

### 5.3 数据可解释要求

Agent 的重要回答应能说明使用了哪些数据。

最低要求：

- 说明关键数据来源，例如本地日线、持仓记录、选股结果、研究报告、历史记忆。
- 说明关键数据日期，例如最新交易日、报告日期、复盘日期。
- 说明数据缺口，例如缺少资金流、缺少公告缓存、日线不足 114 根。

示例：

```text
数据依据：
- 本地日线：最新交易日 2026-05-29，已计算知行多空线
- 个股工作台：存在 1 条跟踪状态
- 历史记忆：命中 2 条关于 000001.SZ 的判断
- 外部数据：资金流缓存缺失
```

## 6. Agent Skill 定义

### 6.1 Skill 的含义

本文中的 Skill 指 AlphaAgents 内部的投研能力模块，不是 Codex 的开发技能。

一个 Skill 是 Agent 面向用户问题时可调用的一组能力说明。

Skill 应包含：

- 能力名称。
- 能解决的问题。
- 需要使用的数据源。
- 可能调用的 tools。
- 输出格式。
- 风险边界。
- 示例问题。

### 6.2 Skill 与 Tool 的区别

Tool 是底层函数调用能力。

例如：

- `get_stock_detail`
- `get_positions`
- `run_selection`
- `recall_memory`

Skill 是面向用户意图的能力组合。

例如：

- 个股诊断 Skill 可能组合 `get_stock_detail`、`recall_memory`、外部数据查询。
- 今日摘要 Skill 可能组合 `get_positions`、`get_daily_summary`、`get_review_history`。

Agent 不应把 tools 直接暴露给用户理解，而应以 Skill 的形式解释“我能帮你做什么”。

## 7. 当前应具备的基础 Skills

### 7.1 今日摘要 Skill

用户问题示例：

- 今天有什么值得关注？
- 今天持仓有没有风险？
- 今天选股结果怎么样？

应使用数据：

- 持仓记录。
- 个股提醒。
- 最新选股结果。
- 最新日报。
- 最新复盘案例。
- Agent 记忆中的重点关注股票。

输出要求：

- 持仓风险。
- 新候选机会。
- 需要复盘的异常点。
- 数据缺口。

### 7.2 个股诊断 Skill

用户问题示例：

- 帮我看一下 000001。
- 这只股票现在有没有破位？
- 这个票我还能继续观察吗？

应使用数据：

- 本地日线和指标。
- 知行多空线、短期趋势线、KDJ、VOL/MACD。
- 个股工作台记录。
- 外部估值、资金流、公告、新闻缓存。
- 历史决策记忆和个股印象。

输出要求：

- 当前趋势状态。
- 风险提醒。
- 与历史判断的关系。
- 关键数据依据。
- 数据缺口。
- 后续观察点。

### 7.3 策略选股 Skill

用户问题示例：

- 跑一下当前策略。
- 当前策略为什么选出这些股票？
- 策略条件现在是不是太宽？

应使用数据：

- 本地行情数据。
- 策略参数。
- 股票和板块元数据。
- 最新选股结果。

输出要求：

- 当前策略条件摘要。
- 选股结果数量。
- 典型命中原因。
- 典型排除原因。
- 是否存在数据不足。

### 7.4 历史回顾 Skill

用户问题示例：

- 我之前为什么放弃 000001？
- 上次复盘这个票是什么结论？
- 我最近经常错过什么类型的机会？

应使用数据：

- Agent 决策记忆。
- 个股印象。
- 操作记录。
- 复盘案例。
- 会话历史摘要。

输出要求：

- 历史判断。
- 时间和来源。
- 当时依据。
- 当前状态是否发生变化。

### 7.5 复盘沉淀 Skill

用户问题示例：

- 帮我记录这次复盘。
- 把这个判断沉淀成案例。
- 更新一下我对这只股票的印象。

应使用数据：

- 当前个股上下文。
- 用户输入。
- 历史操作记录。
- 复盘案例。
- Agent 记忆。

输出要求：

- 先生成待确认内容。
- 用户确认后再写入。
- 写入后说明保存位置。
- 不允许模型自动静默写入。

## 8. 现有功能串联要求

### 8.1 从用户问题到回答的基础链路

Agent 的基本处理链路应为：

```text
用户问题
  -> 识别意图和 Skill
  -> 判断需要哪些数据
  -> 调用已有 tools
  -> 汇总数据和缺口
  -> 生成回答
  -> 必要时生成待确认写入动作
  -> 对话结束后沉淀记忆
```

### 8.2 Chat 与右侧投研面板联动

Chat 不应孤立存在。

基本要求：

- 用户问到某只股票时，右侧 K 线和个股工作台应切换到对应股票。
- Agent 调用个股诊断相关工具时，应刷新右侧提醒和工作台状态。
- 用户点击筛选结果或案例中的股票时，Chat 当前上下文应能感知当前股票。

### 8.3 工作流数据串联

已有功能在 Agent 中应形成以下闭环：

```text
数据同步
  -> 本地行情和元数据更新
  -> 策略选股
  -> 候选结果进入右侧面板
  -> 用户在 Chat 中询问个股
  -> Agent 调用个股诊断和历史记忆
  -> 用户确认记录操作/复盘
  -> 复盘沉淀为案例和记忆
  -> 后续 Chat 可回顾
```

## 9. 用户可见能力说明要求

用户进入 Chat 时，应能看到 Agent 当前能力。

最低要求：

- 展示可用 Skill 列表。
- 每个 Skill 有一句自然语言说明。
- quick action 应对应真实 Skill，而不是随意写死的提示语。
- 用户可以问“你能做什么”，Agent 应基于当前 Skill 列表回答。

示例：

```text
我现在可以帮你：
1. 今日摘要：汇总持仓风险、选股结果和日报。
2. 个股诊断：结合 K 线、提醒、外部数据和历史记忆分析单只股票。
3. 策略选股：运行当前知行趋势策略并解释命中原因。
4. 历史回顾：回顾你过去对某只股票的判断和复盘。
5. 复盘沉淀：生成待确认的操作记录、复盘案例或股票印象。
```

## 10. 新增 Skill 的规范

后续新增 Skill 时，必须先补充 Skill 定义，再实现工具或 UI。

Skill 定义至少包含：

```text
id:
name:
description:
intents:
data_sources:
tools:
output_requirements:
risk_boundary:
examples:
```

新增 Skill 的流程：

1. 先确认是否能复用现有数据和 tools。
2. 如果需要新数据，先定义数据来源和缺口处理方式。
3. 如果需要新 tool，明确 tool 是否只读、是否写入、是否需要确认。
4. 补充测试，验证该 Skill 会使用正确数据。
5. 更新用户可见能力说明。

## 11. 验收标准

本阶段完成时，应满足以下标准：

1. 用户能清楚知道 Agent 当前有哪些能力。
2. Agent 回答个股、今日摘要、历史回顾类问题时，会优先使用项目内数据。
3. 重要回答能说明数据来源和数据缺口。
4. 本地数据、a-stock-data、工作流数据、Agent 记忆之间的职责清晰。
5. 写入型能力必须先生成待确认内容，不得静默写入。
6. 现有功能能通过 Chat 和右侧面板形成投研闭环。
7. 新增 Skill 有统一定义规范，不再随意堆工具或 prompt。

## 12. 当前优先级

短期优先级：

1. 梳理并固化当前 Skill 列表。
2. 梳理每个 Skill 应使用的数据源和 tools。
3. 在 Agent 回答中显式展示数据来源和缺口。
4. 让 Chat 欢迎区和“你能做什么”回答来自同一份能力定义。
5. 检查现有 tools 是否覆盖当前基础 Skills，避免重复新增功能。

暂缓事项：

- 新页面。
- 新数据供应商。
- 多 Agent 协作。
- 自动交易。
- 复杂自主规划。
